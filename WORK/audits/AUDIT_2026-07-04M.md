## Headline

Retail-first queue item #1 is Built and verified.

Finder now has a read-only backend capabilities contract that can become the source of
truth for setup, settings, shell navigation, and demo business-type switching. This does
not implement the UI switcher or paid entitlement enforcement yet.

## What changed

- Added `SettingsService.getCapabilities(auth)`.
- Added top-level `GET /api/v1/capabilities`.
- Added settings alias `GET /api/v1/settings/capabilities`.
- Capabilities response now reports:
  - tenant id;
  - authenticated user role, scopes, store scope, explicit permissions, and all-access status;
  - current business type and whether it came from stored profile or retail default;
  - subscription plan summary from `subscriptions`, with starter fallback;
  - honest entitlement placeholder: `enforced: false`;
  - effective feature flags, including account mode derived from enabled modules;
  - enabled modules with `defaultEnabled`, `source`, and disabled reason;
  - required fields and workflows for the selected business profile;
  - module groups, core modules, and available business types.
- Fresh tenants default to the retail business pack.
- Manual module flags override business-pack defaults and are reported as manual overrides.
- `POST /api/v1/settings/business-profile` now accepts every current
  `BUSINESS_BUNDLES` key instead of the stale short list.

## Honest limitations

- Plan-to-module entitlement enforcement is still not built.
- Business-type impact preview is still not built.
- Setup/settings/demo switcher UI still needs to consume this endpoint.
- Required fields and workflows are contract scaffolding, not full database-enforced
  validation for every business type.
- Only retail is allowed to move toward Built/verified status now; non-retail packs are
  still preview/partial until their own future gates pass.
