## What Changed

The Redis-backed IP and tenant rate limiters no longer use fixed-window keys such as `rl:<client>:<bucket>` or `trl:<tenant>:<bucket>`.

They now use a Redis sorted-set rolling window:

- remove hits older than `now - windowMs`
- count remaining hits with `ZCARD`
- deny when count is already at capacity
- add the current request as a unique member
- expire the key after the rolling window

This closes the boundary burst problem where a client could spend a full bucket at the end of one fixed window and immediately spend another full bucket at the start of the next window.

The Redis return handling was also made explicit: only `1` or `"1"` is treated as allowed. This avoids JavaScript truthiness accidentally treating Redis string `"0"` as allowed.

The frontend production build was also unblocked by disabling Next's separate webpack build worker in `web/next.config.mjs`. On this machine with Node 24, the worker path repeatedly idled forever at `Creating an optimized production build ...`; building in-process lets the same production build complete. During diagnosis, `web/node_modules/js-tokens/package.json` was also found to hang on read and was replaced locally without keeping package.json or lockfile changes.
