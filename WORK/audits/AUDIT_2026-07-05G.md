## Drift bug fixed

The real `POST /api/v1/settings/business-profile` required `businessType` and
silently ignored `moduleFlags` — while the mock accepted `{ moduleFlags }` alone.
Consequences on the real backend:

- The Business Profile page's per-module toggles (and the old modes page before it)
  returned **400** — module toggles have only ever worked against the mock.
- The only way to change modules was a full type switch, which **resets every manual
  override** back to bundle defaults.

## What shipped

### Backend (`src/modules/settings/routes.ts`, `service.ts`, `settings.test.ts`)

- `POST /settings/business-profile` now accepts either or both shapes:
  `{ businessType, enabledModules? }` (type switch — bundle reset as before) and
  `{ moduleFlags: { [key]: boolean } }` (**delta** update: named modules only, other
  flags and the business type untouched; bare or `module:`-prefixed keys; core modules
  cannot be disabled). Empty body → 400.
- Every change is audit-logged via the shared `writeAudit` helper with the acting
  user id: `business_profile.type_changed` (before/after type) and
  `business_profile.modules_changed` (per-module to-states). Rows land in the same
  `audit_log` the `/audit-log` page reads.
- New real-Postgres tests (settings suite now 23/23): delta toggle leaves every other
  module and the business type untouched; prefixed keys accepted; empty body 400;
  audit rows carry actor, timestamp, and the type diff.

### Web (`settings/modes/page.tsx`, `mockHandlers.ts`)

- **Recent changes** section on the Business Profile page: reads
  `GET /api/v1/audit-log?resource_type=business_profile`, shows a human description
  ("Business type: retail → wholesale", "Modules: loyalty disabled"), the actor, and
  a timestamp; refreshes after each switch/toggle. Hidden when there's no history.
- Mock parity: the mock business-profile handler now records the same audit events
  into the shared `/audit-log` feed, so the section works identically in demo mode.
