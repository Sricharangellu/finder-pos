# Ascend - Engineering Audit 2026-07-05B

## Scope

Backend operational readiness for the real backend deployment path.

Claim: Codex session M in `WORK/LOCK.md`.

This audit covers backend ops checks, deploy readiness wiring, metrics exposure, CORS
defaults, and production-mode database SSL configurability. It does not rotate
production demo credentials, edit production database data, build frontend UI, or add
new business modules.

## Verdict

Backend infra readiness is **Built and verified as a deploy gate**.

Ascend as a whole is still **not deployment-ready** until the remaining production
actions are complete: demo credential rotation, production environment confirmation,
authorized metrics configuration, Redis-backed operational services if required for
scale, Stripe/webhook verification, backup/restore proof, and the rest of the
retail-first release gates.

## What changed

- Added `scripts/ops-check.ts`, exposed as `npm run ops:check`.
- Wired `scripts/deploy.sh` to run the richer backend operational readiness check
  after backend deployment instead of only curling `/readyz`.
- Closed production `/metrics` by default when `METRICS_TOKEN` is missing.
- Kept authorized `/metrics` scraping available with `Authorization: Bearer <token>`.
- Added production warnings for missing `METRICS_TOKEN` and `WEBHOOK_SECRET_KEY`.
- Expanded the default backend CORS allowlist for the known deployed frontend URLs.
- Added `PG_SSL` support so production-mode checks can run against local/CI Postgres
  without changing the production default of SSL enabled.
- Added focused gateway ops tests for metrics behavior in production mode.

## Live truth before this push

`https://finder-pos-backend.vercel.app` answered `/healthz`, `/readyz`, and auth-bound
routes correctly, but failed the new readiness checker on two production drift points:

- The deployed frontend origin did not receive the expected CORS allow header.
- `/metrics` was publicly readable without a bearer token.

The code changes close those classes of drift when deployed. Metrics scraping is not
fully operational until `METRICS_TOKEN` is set in the production backend environment.
If the token is absent, the safer production behavior is now `503 metrics_unconfigured`.
