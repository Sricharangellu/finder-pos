## What shipped (`3fa91e2`)

The frontend now renders from the business-pack control plane instead of hardcoded
business-type assumptions — the consumption half of the capabilities contract:

- **`web/contexts/CapabilitiesContext.tsx` (new)** — the tenant layer of the
  four-layer model. Single fetch of `GET /api/v1/capabilities`; exposes
  `moduleEnabled(key)` / `routeEnabled(href)` / full payload / `refresh()`;
  5-minute sessionStorage cache; fail-open while loading or on error so a
  capabilities outage can never blank the navigation.
- **`web/hooks/useModuleFlags.ts` rewired** (same signature, 5 consumers untouched).
  **Real-backend bug fixed:** the old implementation read raw
  `/settings/feature-flags`, which misses business-pack DEFAULTS — a fresh real
  tenant has no module flags stored, so the nav collapsed to core-only. Enabled
  modules now come from capabilities, where core + pack defaults + manual overrides
  are resolved server-side.
- **`EnterpriseShell`** — four-layer nav filtering: explicit `moduleGate`, then
  tenant route gating (a nav href owned by a disabled capabilities module hides,
  covering the whole tree without annotating every item), then the user feature
  gate. Unknown routes stay visible (conservative).
- **`settings/modes` page** — the hardcoded 8-entry `MODES` list is gone. Now a
  capabilities-driven **Business Profile / Plan & Modules** view: current business
  type + plan + entitlements note; a business-type switcher that previews the
  impact (`GET /capabilities/impact` — modules added/removed) before applying via
  `POST /settings/business-profile`; per-module toggles with source badges
  (core / business pack / manual override / not in pack) and disabled reasons,
  grouped by the registry's module groups.
- **Mock parity** — `GET /capabilities` (+ `settings/capabilities` alias + both
  `impact` paths) added to the MSW layer, derived from the same in-memory bundle
  state the business-profile handlers mutate, so mock and real modes render the
  same shell and settings.
