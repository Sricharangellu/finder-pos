# AUDIT 2026-07-04I - Product Variant Atomicity

## Scope

Claim: product lifecycle hardening for catalog variant assignment and generation.

Touched code:

- `src/modules/catalog/service.ts`
- `src/modules/catalog/catalog.test.ts`

No frontend app code, e2e specs, report/EOD files, orders, payments, outlets, or smoke
script files were changed.

## What Changed

Multi-child variant operations are now database-atomic:

- `assignVariants()` runs every child assignment inside one tenant-scoped transaction.
- `generateVariants()` creates the whole variant matrix inside one tenant-scoped transaction.
- If any child fails validation or SKU allocation fails midway, all product row changes
  from that operation roll back.

Domain events are also safer:

- `CatalogService.create()` and `CatalogService.update()` now support deferring their
  immediate event publish.
- Bulk variant assignment and generation suppress events while the transaction is open.
- `product.updated` / `product.created` events publish only after the database commit.

This fixes the earlier risk where the first child could be linked before a later child
failed. It also prevents an event-handler deadlock observed during verification: publishing
`product.updated` inside the open variant transaction caused the sync event handler to try
to open another DB transaction while the test harness pool was pinned to one connection.

## Remaining Gaps

This moves one product lifecycle gap from partial to built and verified, but product
readiness still needs:

- searchable parent/master product picker in the frontend
- UI for variant matrix generation and preview
- real-backend Playwright proof for master create -> child create -> variants view
- product edit, supplier linking, price history, inventory stock, and ecommerce publish
  workflows connected into one verified lifecycle

The active e2e core-flow triage remains owned by the Antigravity team lock, so this
session intentionally did not touch `web/e2e/**` or run local e2e servers.

## Verdict

Product variant bulk writes are now **Built and verified** for database atomicity and
post-commit event publication. The full application is still **Not production-ready**
until the real-backend e2e core flows and broader product lifecycle are proven.
