## Headline

Production frontend deploys can no longer silently ship the MSW/mock-mode demo build.

The app source still defaults `NEXT_PUBLIC_MOCK` to `"true"` in `web/next.config.mjs`,
which is useful for local demos and safe previews. That default was dangerous for the
current production deploy path because `scripts/deploy.sh` stages a fresh frontend build.
If the deploy environment did not explicitly override `NEXT_PUBLIC_MOCK`, production
could build the demo UI against MSW instead of the live backend even after the real
backend e2e core flows were proven green.

## What changed

- `.github/workflows/deploy-prod.yml` now sets `NEXT_PUBLIC_MOCK: "false"` for the
  production deploy job.
- `scripts/deploy.sh` now defaults production frontend deploys to
  `NEXT_PUBLIC_MOCK=false`.
- `scripts/deploy.sh` now refuses production frontend deploys if `NEXT_PUBLIC_MOCK` is
  explicitly set to anything other than `false`.
- Non-production deploys (`DEPLOY_ENV=testing|dev`) still default to mock mode, preserving
  demo/preview behavior.
- Existing runtime demo mode on the live frontend remains available through the app's
  current `?demo=1` / localStorage demo path.
- `WORK/FORWARD_PLAN.md` now defines Ascend as a modular business operating platform,
  not only a retail POS.
- `WORK/FORWARD_PLAN.md` now documents the business-pack architecture: plan, business
  type, entitlements, and permissions are separate layers; business type selection
  installs defaults but must not fork the app.
- `WORK/FORWARD_PLAN.md` now records the current business-mode implementation as
  **Partial**: module registry, business profile, feature flags, mode toggles, nav gates,
  and permissions exist, but required-field rules, workflow constraints, plan
  entitlements, impact previews, and complete backend enforcement are not finished.
- `WORK/FORWARD_PLAN.md` now tells developers and company admins how business-type
  changes should be visible: developer matrix from the pack registry, and company-facing
  module/pack impact view before switching business type.

## Honest status

This is the right production safety move. It does not remove demo mode, and it does not
pretend the broader staging decision is finished.

Still open:

- Choose a dedicated staging database/deploy target.
- Decide whether `web/next.config.mjs` should keep mock-on as the source default for
  local demos or flip to real-backend by default now that the core real-backend e2e
  suite is green.
- Keep proving real behavior with production builds and real backend smoke/e2e gates;
  mock-mode screenshots are not deployment evidence.
- Business type switching is not complete. The current app can toggle module bundles and
  feature flags, but it still needs real entitlements, required-field constraints,
  workflow presets, permission templates, audit history, and an impact/preview endpoint
  before it should be treated as production-grade business-pack switching.

## Verdict

Better than flipping the source default immediately. Production is now protected first,
while local/demo preview behavior stays intact until the staging environment decision is
made.

The product scope correction is also the right direction: one shared operating engine,
business packs on top, no duplicated vertical mini-apps.
