## What Changed

Product creation is now connected to the existing catalog schema instead of only
posting a shallow SKU/name/price payload. The new product page can create:

- standalone products
- master products with an implicit zero price
- child variant products linked to a parent product id and variant label

The create payload now includes catalog, pricing, inventory-rule, physical,
operations, ecommerce, barcode, vendor UPC, image URL, tax, and variant fields
already supported by the backend product model.

Variant relationships are stricter on the backend:

- a child variant cannot be used as a master product
- a master product that already has children cannot be made a child of another product
- self-parenting remains rejected
- direct child create and parent clearing are validated
- bulk assignment can set a variant label and returns the updated variant list
- child variants can be unlinked through `DELETE /api/v1/catalog/:id/variants/:childId`
- master products can generate a variant matrix from attribute groups through
  `POST /api/v1/catalog/:id/variants/generate`

The variants tab now links from a master product directly into the child-product
create flow with the parent id prefilled.

## Remaining Gaps

This is not full enterprise product lifecycle completion. It strengthens the core
create and variant graph paths, but these product areas still need development:

- variant assignment and generation should eventually be made atomic so a multi-child
  operation cannot partially apply if a later child fails validation
- the frontend still asks for parent product id instead of providing a searchable
  master-product picker
- generated variant matrices are backend-only; there is no UI for selecting
  attribute groups and previewing combinations
- there is no end-to-end Playwright proof for create master -> create child ->
  view variants against the real backend
- product edit, inventory stock, supplier linking, price history, and ecommerce
  publish workflows are still separate incomplete flows

## Verdict

The product create and master/child variant direction is correct, but the product
domain is not deployment-complete by itself. This work closes important data-model
and UI gaps without expanding into unrelated modules. The project still needs a real
core e2e pass and product lifecycle coverage before it should be called production
ready.
