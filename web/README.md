# Ascend — Web (Frontend)

Next.js 14 App Router · TypeScript · Tailwind CSS · MSW mocks · openapi-typescript

---

## Quick start

```bash
cd finder-pos/web

# 1. Install dependencies (also copies the MSW service worker to public/)
npm install

# 2. Copy the example env file
cp .env.local.example .env.local

# 3. Start the dev server (MSW mocks are auto-enabled in development)
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

In dev mode the login form is pre-filled with `cashier@ascend.dev`.
Use any password (enter `wrong` to test the error state).

## Regenerating the API client

Whenever `contracts/openapi.yaml` changes (the Backend agent publishes a new version), regenerate:

```bash
cd finder-pos/web
npm run generate:client
# Commit the updated api-client/types.ts
```

The generator target is `api-client/types.ts`. **Do not hand-edit this file** — your changes will be overwritten.

## How MSW mocks work

1. During development (`NODE_ENV=development`) the `MockWorkerInit` component (rendered in `app/layout.tsx`) dynamically imports `mocks/browser.ts` and starts the MSW service worker.
2. The service worker intercepts every `fetch` call to `/api/v1/*` and returns mock responses from `mocks/handlers.ts`.
3. In tests, `mocks/server.ts` provides an MSW Node.js server that `tests/setup.ts` starts before the test suite.

**To switch an endpoint from mock to live** (when the Backend ships a real route):

Remove the corresponding handler from `mocks/handlers.ts`. The fetch will now pass through to the real backend (proxied via `next.config.ts` rewrites in dev).

**To force mocks in production** (e.g. demo environments):

```bash
NEXT_PUBLIC_MOCK=true npm run build
```

## Feature flags

Flags are fetched from `GET /api/v1/flags` every 60 seconds.
Unknown flags **default to `false`** — new UI is hidden until the backend enables it.

```tsx
import { useFlag } from "@/flags";

function MyComponent() {
  const showNewFeature = useFlag("my_feature_flag");
  if (!showNewFeature) return null;
  return <NewFeature />;
}
```

## Auth flow

1. User submits the login form → `POST /api/v1/auth/login`
2. On success: access token stored in memory, refresh token in `sessionStorage`
3. On every API call: `Authorization: Bearer <accessToken>` header added by `api-client/client.ts`
4. On page reload: `useAuth` calls `silentRefresh()` → `POST /api/v1/auth/refresh` using the stored refresh token
5. On logout: `POST /api/v1/auth/logout`, then both tokens cleared

The access token is **never** written to `localStorage` to limit XSS exposure.

## Accessibility

All design-system components target WCAG 2.1 AA:

- Minimum 44×44 px touch targets (WCAG 2.5.5) via `min-h-[44px] min-w-[44px]`
- Visible `:focus-visible` ring (3 px brand colour)
- All interactive elements have accessible labels
- Error messages use `role="alert"` (assertive) or `role="status"` (polite) as appropriate
- Skip-to-content link in `app/layout.tsx`
- Colour contrast ≥ 4.5:1 for all text/interactive elements

## Money handling

**Never use floats for money.** All monetary values travel as integer cents.
Use `lib/money.ts` for formatting and arithmetic:

```ts
import { formatMoney, parseToCents, calcChange } from "@/lib/money";

formatMoney(1099)          // "$10.99"
parseToCents("$10.99")     // 1099
calcChange(2000, 1599)     // 401  (change due)
```
